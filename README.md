# SQL Data Warehouse — Medallion Architecture on Azure SQL

A small but complete data-warehouse implementation following the
**Medallion architecture** (Bronze / Silver / Gold) on **Azure SQL Database**.
Source data is two CSV-based systems — a CRM extract and an ERP extract —
that are progressively cleaned, conformed, and modeled into a Kimball
star schema for analytics.

📄 **Read the full write-up:** [Portfolio article — *A Data Warehouse Two Systems Can Agree On*](outputs/portfolio_dw_article.md)

## Architecture

```
                +-------------------+      +-------------------+      +-------------------+
  CSV files --> |  bronze (raw)     | -->  |  silver (clean)   | -->  |  gold (modeled)   |
                |  6 tables, NVARCHAR|     |  6 tables, typed  |     |  3 views (star)   |
                +-------------------+      +-------------------+      +-------------------+
                  TRUNCATE + LOAD             TRUNCATE + INSERT          CREATE VIEW
                  (BULK INSERT)               (silver.load_silver)        (gold DDL)
```

* **Bronze** — raw, as-ingested data. No constraints, all `NVARCHAR(50)` for
  strings, dirty INT-encoded dates kept as INT. Full refresh.
* **Silver** — typed, deduplicated, cleansed. Codes expanded (e.g. `M` →
  `Male`), dates parsed, arithmetic reconciled, natural keys harmonized
  across CRM and ERP. Each row carries a `dwh_create_date` audit column.
* **Gold** — Kimball star schema exposed as views. `dim_customers`,
  `dim_products` (current versions only), and `fact_sales` joined via
  surrogate keys generated by `ROW_NUMBER()`.

## Repository layout

```

  init_database.sql          -- create the three schemas (idempotent)
  bronze/
    ddl_bronze.sql           -- create the 6 bronze tables
    proc_load_bronze.sql     -- BULK INSERT from Azure Blob (reference)
```

## How to run end-to-end

Run each script against the Azure SQL database in this order. The procs
are `CREATE OR ALTER`, so re-running them is safe.

1. **Schemas** (one-time)
   ```sql
   :r scripts/init_database.sql
   ```
2. **Bronze tables** (one-time, or whenever DDL changes)
   ```sql
   :r scripts/bronze/ddl_bronze.sql
   ```
3. **Bronze data** — load CSVs into bronze. On Azure SQL this requires
   either:
     * The `BULK INSERT` proc in `scripts/bronze/proc_load_bronze.sql`
       wired to an Azure Blob external data source, OR
     * A one-shot import via the Azure Data Studio / SSMS Import Wizard
       (what was done initially for this project).
4. **Silver tables**
   ```sql
   :r scripts/silver/ddl_silver.sql
   ```
5. **Silver load** (run after every bronze refresh)
   ```sql
   EXEC silver.load_silver;
   ```
6. **Gold views** (one-time, or whenever the model changes)
   ```sql
   :r scripts/gold/ddl_gold.sql
   ```
7. **Quality checks** — every query should return zero rows
   ```sql
   :r tests/quality_checks_silver.sql
   :r tests/quality_checks_gold.sql
   ```

## Cleansing rules applied in silver

| Table | Key transformations |
|---|---|
| `crm_cust_info` | `TRIM` names; `M`/`F` → Male/Female; `M`/`S` → Married/Single; dedupe on `cst_id` keeping the latest `cst_create_date`; drop NULL ids |
| `crm_prd_info` | Split `prd_key` into `cat_id` (first 5 chars, `-` → `_`) + `prd_key`; `ISNULL(cost,0)`; `M/R/S/T` → Mountain/Road/Other Sales/Touring; `prd_end_dt = LEAD(prd_start_dt) - 1 day` |
| `crm_sales_details` | INT YYYYMMDD → `DATE` (sentinel `0` / wrong length → NULL); recompute `sls_sales` when missing or inconsistent with `qty * |price|`; derive `sls_price = sales / NULLIF(qty,0)` when invalid |
| `erp_cust_az12` | Strip `NAS` prefix from `cid`; future birthdates → NULL; gender normalized to Male / Female / n/a |
| `erp_loc_a101` | Strip `-` from `cid`; `DE` → Germany, `US`/`USA` → United States; blank → n/a |
| `erp_px_cat_g1v2` | Pass-through (already clean) |

## Star schema (gold)

```
                        +----------------------+
                        |   dim_customers      |
                        |  PK customer_key     |
                        +----------+-----------+
                                   |
                                   | customer_key
                                   v
+---------------------+    +---------------+    +---------------------+
|   dim_products      |--->|  fact_sales   |<---|  (date is degenerate|
|  PK product_key     |    |               |    |   via order_date)   |
+---------------------+    +---------------+    +---------------------+
        ^ product_key
```

`fact_sales` grain: one row per sales-order line.
